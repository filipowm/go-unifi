# Test Lead / SDET consolidating area reviews for the go-unifi maintainer — go-unifi review (target 1.11.0)

> Generated by a two-persona multi-agent review (5 architect + 3 test-lead area reviewers, synthesized). Findings prefixed ✅ were independently re-verified against source by the lead before hand-over.

## Executive summary

**Overall health: solid foundations, dangerous blind spots.** The test suite is genuinely strong where it has been written — HTTP request machinery, interceptors, error formatting/parsing, validation
predicates, and codegen type-inference are well covered with disciplined table-driven, testify-based, `t.Parallel` tests against real `httptest` round-trips. The team clearly knows how to inject
seams (the `UnifiVersionProvider` mock and the decomposed `newClientFromConfig` helpers are exemplary). But coverage is unevenly distributed and the highest-value outputs are the least tested.

**Biggest themes.** (1) **The two most load-bearing outputs of the project are unverified:** every hand-written `unifi/` resource wrapper (user/sites/device/setting CRUD — the exact layer contributors
are told to test) sits at 0%, and the codegen resource templates (`api.go.tmpl`/`apiv2.go.tmpl`) that emit all `*.generated.go` source are asserted by nothing. (2) **Testability friction is the root
cause of the wrapper gap:** `determineApiStyle` hard-wires its own HTTP client and runs inside every constructor, forcing tests to either stand up a full server or swallow construction errors, and the
per-resource `customizeBaseType`/`customizeResource` special-casing (backwards-compat critical) is near-untested. (3) **Security guards are unpinned:** the decompression-bomb limiter (`copyWithLimit`)
and the zip path-traversal check (`sanitizeExtractedPath`) only ever exercise their happy paths. (4) **Network coupling makes the codegen suite slow, flaky, and offline-hostile** — multiple tests
download a multi-MB controller `.deb` from Ubiquiti's CDN with no `-short`/build-tag guard. (5) **There is also real bloat:** duplicated type-inference tests and over-engineered registry tests.

**What to tackle first.** Add a golden/snapshot test for the codegen resource templates (TEST-01) — it is the single biggest correctness risk and cheap to land. In parallel, build a shared
wrapper-test harness on the existing `newNewStyleClient`+`HttpRoundTripperProvider` seam (TEST-03) to unblock the 0%-coverage wrappers, the never-asserted `ErrNotFound` contract, and the untested
per-resource codegen customizations (TEST-04/05). Then pin the two security guards (TEST-02) and gate the live-network codegen tests behind `testing.Short()` (TEST-08). Delete the confirmed dead
duplicated block in `customizeBaseType` (TEST-06) and trim the redundant tests (TEST-17) to keep the suite honest.

## Strengths

- HTTP request machinery is genuinely well covered: requests.go helpers (marshalRequest, buildRequestURL, validateRequestBody, applyRequestInterceptors, overrideHeaders) at 100%, handleResponse 92.9%,
  with focused unit tests covering decode happy-path, decode-error wrapping, zero-ContentLength short-circuit, and nil-respBody paths — and the decode/error logic was deliberately extracted so it
  tests with hand-built *http.Response values and zero network.
- Error formatting/parsing is thoroughly table-tested (single/multi/no-detail rendering, V1 and V2 decode branches, 2xx early return), pushing parseApiV2Error and HandleError to 100%.
- interceptors.go is at 100% across all three interceptors via real httptest round-trips, with TestCsrfHandling proving the two-request token-capture-then-replay flow.
- Codegen version resolution is correctly abstracted behind the UnifiVersionProvider interface with firmwareUpdateApi injected, fully mocked via httptest (assertLatestVersionUsingProvider) — error
  paths (HTTP 500, invalid JSON, missing firmware, nil version, invalid URL) all covered. This is the right network-seam pattern and proves the team can do it.
- newClientFromConfig was deliberately decomposed into small pure-ish helpers (resolveLogger, buildHTTPClient, resolveCredentials, buildInterceptors, resolveErrorHandler), each independently
  unit-tested — a strong, test-driven factoring of a usually-monolithic constructor.
- Codegen type-inference and validation predicates are rigorously tested: TestFieldInfoFromValidationDetails/TestNormalizeValidation pin the numeric/float/int/IP-octet/bool branches, and
  validation_test.go fuzzes IPv4/IPv6/IP/numeric matchers across ~20 permutations each. customize.go behavior (the _all keyword, ifFieldType gating, composite-processor ordering) is well-pinned.
- The Logger interface + noopLogger gives a clean time/IO-free seam in the unifi package (no global logger there), and ClientConfig exposes the right injection points (HttpRoundTripperProvider,
  HttpTransportCustomizer, ClientInterceptor, ResponseErrorHandler) so wrappers are exercisable against httptest without touching production HTTP.
- Tests broadly follow project conventions: table-driven map[string]struct with t.Parallel on outer and subtests, testify assert/require, external unifi_test package where the public API is exercised.
  setting_test.go is a model registry-contract test (pins factory keys to concrete types, fresh-instance semantics, size lockstep, unknown-key error).
- Security-relevant guards exist in production with clear intent comments (copyWithLimit decompression-bomb defense, sanitizeExtractedPath zip-slip check) even where currently under-tested — the
  intent is documented and the seams are present.

## Findings (detail)

### TEST-01 [P0] Codegen resource templates (api.go.tmpl / apiv2.go.tmpl) have ZERO tests — the core generated output is unverified

- **Category:** Coverage · **Effort:** M · **Risk:** Low
- **Locations:** codegen/resources.go:187; codegen/api.go.tmpl:1; codegen/apiv2.go.tmpl:1; codegen/generator_test.go:166
- **Problem:** The single most load-bearing output of the project — the generated <resource>.generated.go source that every consumer depends on — is never asserted in any test.
  Resource.GenerateCode() (resources.go:187, only 66.7% covered) renders api.go.tmpl/apiv2.go.tmpl, but no test calls it on a real Resource. The only GenerateCode() test renders the trivial client
  template, and TestGenerateCodeFromFields runs the full pipeline with an EMPTY fields dir, so zero resources render and both resource templates are exercised by nothing. A change to struct layout,
  the UnmarshalJSON typecast block, Setting-vs-non-Setting CRUD endpoints, the Device 'stat/' vs APGroup vs 'rest/' path logic, or custom-unmarshal field aliasing would produce broken or wrong Go and
  every test would still pass. The entire V2 branch (IsV2 -> apiv2.go.tmpl, different ApiV2Path endpoints, no Meta wrapper) is completely untested — the missing 33% of GenerateCode IS the IsV2 path.
- **Proposals:**
    - Add a golden/snapshot test: build a Resource via NewResource with a representative field set (string, int+emptyStringInt, float64, bool, nested struct, array, a custom-unmarshal field), call
      GenerateCode(), and assert against a checked-in .golden file with a -update flag. Because GenerateCode runs format.Source, golden output is gofmt-stable and diffs are readable; format.Source
      also fails the test on any syntactically invalid Go, catching template breakage for free.
    - Add at least one V2 resource case (resource.V2=true) so apiv2.go.tmpl and the IsV2 branch are covered, asserting the ApiV2Path-based endpoints and the no-Meta response shape — this is the
      cheapest way to lift GenerateCode toward 100%.
    - For low-maintenance pinning of the conditional endpoint logic, prefer focused Contains assertions over a full golden file where a golden file would be noisy: e.g. Setting resource output
      Contains 'const SettingXKey =' and 'GetSettingX'; Device list endpoint Contains 'stat/'; APGroup contains neither 'stat/' nor 'rest/' (pins api.go.tmpl:78/110). Trade-off: golden files catch
      unintended diffs broadly but churn on every template tweak; Contains assertions are stable but only guard the specific lines you name.

### TEST-02 [P1] Security guards untested on the failure path: decompression-bomb limiter and zip path-traversal check only exercise happy paths

- **Category:** Security · **Effort:** S · **Risk:** Low
- **Locations:** codegen/utils.go:60; codegen/download.go:222; codegen/download.go:158; codegen/download.go:233; codegen/download_test.go:113
- **Problem:** Two security-relevant defenses in the codegen extract path have their critical branch untested. copyWithLimit (utils.go:60, the G110 decompression-bomb defense) is only 75% covered and
  its overflow branch (n > maxSize -> 'possible decompression bomb' error) is NEVER exercised. sanitizeExtractedPath (66.7%) is the zip-slip/path-traversal guard, but the only test (
  TestSanitizeExtractedPath_Valid) exercises the happy path; the security branch (!strings.HasPrefix(absFilePath, absDestinationDir) -> 'invalid file path') at download.go:233 is never tested.
  extractZipEntry's error paths (f.Open/os.Create failure, oversize via copyWithLimit) are also untested. A regression that weakens or removes either guard would fail no test — exactly the kind of
  silent security erosion that matters in code parsing untrusted downloaded archives.
- **Proposals:**
    - Unit-test copyWithLimit directly (no network): feed a bytes.Reader larger than a tiny maxSize and assert the 'decompression bomb' error; feed a smaller one and assert success plus the returned
      byte count.
    - Add sanitizeExtractedPath cases with traversal-style inputs ('../../etc/passwd', '/abs/evil', names containing '..') asserting either the sanitized in-dir result or the 'invalid file path' error
      where the guard is meant to trip. Note filepath.Base() already strips directory components so traversal is mitigated, but the explicit guard is the line that must stay covered — the test
      documents the intended behavior.
    - Add an extractZipEntry oversize test: craft a zip entry larger than maxJSONSize and assert the decompression-bomb error propagates as 'unable to write JSON file'.

### TEST-03 [P1] All hand-written unifi/ resource wrappers (user/sites/device/setting CRUD) are at 0% coverage, and the network-probe constructor is the structural cause

- **Category:** Coverage · **Effort:** L · **Risk:** Low
- **Locations:** unifi/user.go:13-194; unifi/sites.go:22-115; unifi/device.go:26-103; unifi/setting.go:71-146; unifi/requests.go:255-278; unifi/api_paths.go:77-82; unifi/client.go:330-346;
  unifi/helpers_test.go:24-28
- **Problem:** Every hand-written resource wrapper — the exact layer the project asks contributors to test (CLAUDE.md: 'put custom logic in the wrapper') — has zero tests, none of it generated.
  GetUserByMAC, CreateUser (nested objects/group response shaping + Meta.error() handling), the stamgr command family (Block/Unblock/Delete/Kick by MAC), OverrideUserFingerprint (DELETE-vs-PUT branch
  on devIdOverride==0), GetSite/GetDevice (linear-search-then-ErrNotFound), CreateSite/DeleteSite/UpdateSite, AdoptDevice/ForgetDevice, and SetSetting/GetSetting (key-match loop + newFields +
  double-unmarshal) all run untested, each containing real branching and request-body shaping that regressions would silently break. There is also no happy-path test proving the {meta,data}
  list-envelope unwrap (ListSites/ListUser/ListDevice) returns the inner slice — the most common real-world path is covered only transitively — and the Put/Delete convenience wrappers are 0%. Root
  cause of the friction: newBareClient always runs determineApiStyle, which probes baseURL over the network, so every wrapper test must stand up a full server or swallow the construction error and
  hand-pin apiPaths via newNewStyleClient.
- **Proposals:**
    - Promote newNewStyleClient (pins apiPaths, skips the probe) to the canonical wrapper-test builder and drive the suite through a custom http.RoundTripper (via HttpRoundTripperProvider) returning
      canned {meta,data} responses — no live server needed, eliminating the swallowed-error foot-gun.
    - Prioritize the branchy wrappers: CreateUser (malformed-group-response error, Meta.error() error, ErrNotFound on len!=1, success), OverrideUserFingerprint (assert DELETE when devIdOverride==0 vs
      PUT otherwise plus the v2 path), GetSite/GetDevice (found vs ErrNotFound), SetSetting/GetSetting (key found, key-not-found ErrNotFound, unknown-key newFields error).
    - Assert outgoing request bodies for command endpoints by decoding r.Body in the handler (BlockUserByMAC posts cmd=block-sta with the mac, DeleteUserByMAC sends a macs array, AdoptDevice sends
      cmd=adopt).
    - Add a TestListSites/TestListUser case returning a multi-element {"data":[...]} body and assert slice length + field values to prove the envelope unwrap, plus a minimal Put/Delete round-trip
      asserting method propagation.

### TEST-04 [P1] Per-resource codegen special-casing (customizeBaseType / customizeResource) is almost entirely untested — backwards-compat-critical and currently at 17.6% / 20.0%

- **Category:** Coverage · **Effort:** M · **Risk:** Low
- **Locations:** codegen/resources.go:402; codegen/resources.go:435
- **Problem:** customizeBaseType (17.6%) and customizeResource (20.0%) are the lowest-covered logic in the package and they encode hand-maintained, backwards-compat-critical special cases: Device gets
  MAC/Adopted/Model/State/Type; User gets IP/DevIdOverride; WLAN keeps the removed-in-v6 WLANGroupID; Setting gets a Key field; SettingUsg keeps MdnsEnabled and rewrites *Timeout fields to
  int/emptyStringInt; SettingGlobalAp rewrites 6E->SixE; SettingMgmt injects the XSshKeys nested struct and a FieldProcessor. None of these branches are asserted by any test — a regression that drops
  Device.State or breaks the SettingUsg timeout rewrite would ship silently and corrupt generated structs for real users.
- **Proposals:**
    - Add a table-driven test over structName -> expected injected base fields: call NewResource(name, path) then customizeBaseType and assert resource.BaseType().Fields contains the expected
      entries (Device: MAC/Adopted/Model/State/Type; User: IP/DevIdOverride; WLAN: WLANGroupID; SettingUsg: Key/MdnsEnabled).
    - Test the FieldProcessor side-effects from customizeResource: SettingGlobalAp on a '6E'-prefixed field -> FieldName becomes 'SixE...'; SettingUsg on 'FooTimeout' -> FieldType=='int' and
      CustomUnmarshalType=='emptyStringInt', and 'ArpCacheTimeout' is NOT rewritten; SettingMgmt -> XSshKeys struct registered in resource.Types and the field retyped.
    - Do this together with TEST-06 so the single intended path (after removing the dead duplicate) is what gets pinned.

### TEST-05 [P1] The ErrNotFound public sentinel contract is never asserted (confirmed zero test references) despite being load-bearing for callers

- **Category:** Coverage · **Effort:** M · **Risk:** Low
- **Locations:** unifi/user.go:25; unifi/user.go:68; unifi/sites.go:48; unifi/device.go:58; unifi/setting.go:93; unifi/sysinfo.go:89
- **Problem:** .claude/rules/go-conventions.md mandates 'return the sentinel ErrNotFound when a list endpoint yields zero items', and callers do errors.Is(err, ErrNotFound) matching — yet a grep of
  unifi/*_test.go confirms ZERO test references to ErrNotFound. The len(Data)!=1 / not-found branch in at least six wrappers is unverified, so a refactor returning a generic error or nil instead of
  ErrNotFound, or one that breaks the sentinel identity across the %w wrap chain, would pass CI. This is a documented public contract with no guard.
- **Proposals:**
    - Inside the TEST-03 wrapper suite, add cases where the server returns an empty (or >1-element) data array and assert require.ErrorIs(t, err, unifi.ErrNotFound) for GetUserByMAC, GetSite,
      GetDevice, GetSystemInfo, and SetSetting/GetSetting.
    - Add one explicit guard asserting the sentinel identity survives wrapping (errors.Is across the %w chain) so a future error-rewording cannot silently break errors.Is matching.

### TEST-06 [P1] Confirmed dead duplicated block in customizeBaseType (pre-switch block byte-for-byte repeated as first switch case)

- **Category:** Maintainability · **Effort:** S · **Risk:** Low
- **Locations:** codegen/resources.go:404-419
- **Problem:** Verified by reading the file: customizeBaseType writes the IsSetting() Key/MdnsEnabled injection twice — unconditionally at lines 404-411 (if resource.IsSetting()) and again identically
  as the first switch case at 413-419 (case resource.IsSetting()). The two blocks are byte-for-byte identical; the second re-assigns the same map keys, so it is harmless at runtime (idempotent map
  writes) — which is exactly why it is dangerous: no test asserts Setting base fields (TEST-04), so the duplication and any future divergence between the two copies is invisible. This is almost
  certainly a bad-merge/refactor leftover, and a maintainer editing one copy and not the other would create a silent inconsistency.
- **Proposals:**
    - Delete the unconditional pre-switch block (lines 404-411) and keep only the case resource.IsSetting(): branch, so Setting handling lives in exactly one place consistent with the Device/User/WLAN
      cases.
    - Land the TEST-04 Setting/SettingUsg assertions in the same change so the single remaining path is locked in and future duplication is caught.

### TEST-07 [P1] Codegen download/extract chain (.deb -> ar -> xz -> tar -> ace.jar) is bound to http.DefaultClient with no seam, so the critical path is only 'covered' opaquely by live-network tests

- **Category:** Testability · **Effort:** M · **Risk:** Medium
- **Locations:** codegen/download.go:59; codegen/download.go:65; codegen/download.go:84; codegen/download.go:107; codegen/main.go:78; codegen/download_test.go:37
- **Problem:** downloadJar uses http.DefaultClient.Do directly (download.go:65) and DownloadAndExtract takes a url.URL rather than an injected client/transport — unlike version.go, which correctly
  hides HTTP behind UnifiVersionProvider and is fully mocked. Consequently downloadJar/openDebDataTar/extractAceJar have NO direct unit tests; their coverage (76.9/69.2/73.7%) exists only because the
  live-network integration tests actually download a real controller .deb during the coverage run. DownloadAndExtract is tested only for its two early-return branches, never the real download+extract
  flow, and the byte-level success path of openDebDataTar/extractAceJar (both already take io.Reader) plus their 'unable to find .deb data file' / 'unable to find ace.jar' error branches go untested.
  This makes the most critical codegen path slow, flaky, and unrunnable in hermetic/offline CI.
- **Proposals:**
    - Inject an *http.Client (or http.RoundTripper) into downloadJar/DownloadAndExtract, defaulting to http.DefaultClient, so an httptest.NewServer serving a tiny hand-built ar(data.tar.xz(ace.jar))
      fixture can drive the full openDebDataTar->extractAceJar->extractJSON chain offline.
    - Exploit the existing io.Reader seam: feed a constructed ar+xz+tar byte stream (built once via a test helper, e.g. github.com/xor-gate/ar writer) to openDebDataTar/extractAceJar to lock the happy
      path, plus a stream missing the data.tar.xz / ace.jar entry to hit the two error branches.
    - Optionally test downloadJar against an httptest server returning a crafted small .deb and a 404 server to cover the StatusCode!=200 branch, removing the live fw-download call entirely.

### TEST-08 [P1] Codegen test suite requires live internet by default (downloads a multi-MB .deb, shells out via `go run .`) with no -short / build-tag guard

- **Category:** Test-design · **Effort:** M · **Risk:** Low
- **Locations:** codegen/main_test.go:71; codegen/main_test.go:89; codegen/main_test.go:114; codegen/main_test.go:133; codegen/main_test.go:151
- **Problem:** Multiple tests hit the real network or shell out with no testing.Short() guard (the only t.Skip in the package is the root-permission one): TestGenerateLatest/TestGenerateDownloadOnly
  call generate(latest), resolving the version from the live fw-update API AND downloading the full unifi_sysvinit_all.deb; TestDefaultVersion/TestHelpFlag/TestInvalidFlag spawn `go run .`
  subprocesses (slow, needs toolchain + module download); TestNonExistentVersion also calls the live API. The suite is therefore slow, flaky, offline-hostile, and dependent on Ubiquiti's CDN/API being
  up and a specific firmware still being downloadable — a poor property for a suite that should be fast and deterministic. (This is the same root coupling as TEST-07, viewed from the suite side.)
- **Proposals:**
    - Gate the network/exec integration tests behind if testing.Short() { t.Skip(...) } or a //go:build integration tag, so `go test -short ./codegen` runs purely offline and fast and CI opts into the
      heavy ones explicitly.
    - Where feasible, redirect the firmware-API dependency through the existing UnifiVersionProvider/httptest seam — generate() already takes firmwareUpdateApi (main_test.go:108 sets it) — pointing
      TestNonExistentVersion-style cases at an httptest server.
    - For the help/invalid-flag CLI tests, prefer invoking the in-process flag-parsing path rather than spawning the Go toolchain per test; keep them but mark them integration if the subprocess form
      is required.

### TEST-09 [P1] determineApiStyle hard-wires its own HTTP client and runs in every constructor — the 302/old-style and API-key-rejection branches are untested and the decision logic is structurally un-isolatable

- **Category:** Testability · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/api_paths.go:67; unifi/api_paths.go:77; unifi/api_paths.go:97-106; unifi/client.go:342
- **Problem:** determineApiStyle (77.3% covered) builds its OWN one-off &http.Client{...} internally (api_paths.go:77) instead of using c.http or an injected round-tripper, and is invoked
  unconditionally from newBareClient (client.go:342). It cannot be unit-tested in isolation, and because it always does a network GET no client can be constructed offline — the root cause forcing the
  swallowed-construction-error pattern. Two non-trivial, correctness/security-relevant branches are untested: the StatusFound (302 -> OldStyleAPI) path, and the guard at lines 104-106 rejecting
  API-key auth against the old-style API ('unable to use API key authentication with old style API'). The only existing api_paths test covers the 500-status error case. Since this method decides the
  auth path, silently breaking either branch ships a confusing user-facing failure.
- **Proposals:**
    - Extract the pure decision into apiStyleFromStatus(status int, isAPIKey bool) (*APIPaths, error) and unit-test all branches (200/302/other x apikey/userpass) with zero network; have
      determineApiStyle call it. This alone covers the currently-missing branches cheaply.
    - Route the HTTP call through an injectable seam — reuse c.http (with a per-call CheckRedirect override via a cloned client) instead of a fresh http.Client — so HttpRoundTripperProvider/httptest
      already in ClientConfig also controls style detection and a 302-returning server can finally exercise the old-style branch.
    - Optionally add a ClientConfig.APIStyle override to skip detection entirely, which also enables offline construction and unblocks the wrapper-test ergonomics in TEST-03.

### TEST-10 [P1] Downstream consumers cannot mock the generated Client interface — no exported mock, no offline constructor seam

- **Category:** Testability · **Effort:** M · **Risk:** Low
- **Locations:** unifi/client.generated.go:11; unifi/client.go:118; unifi/client.go:306; unifi/client.go:326
- **Problem:** The package exports a huge ~400-method Client interface but the only implementation is the unexported struct client, and the only constructors (NewClient/NewBareClient) call
  determineApiStyle and thus require a reachable controller. A downstream consumer cannot construct a real Client without a live/httptest controller, nor substitute a fake (no exported mock;
  hand-rolling a ~400-method stub is impractical). This pushes every consumer toward integration tests and is the single biggest external testability gap. Internally the project sidesteps it with
  package-private helpers external code cannot use.
- **Proposals:**
    - Generate a mock alongside the interface: add a //go:generate moq -out client_mock.go . Client directive (in a hand-written sibling so daily regeneration keeps it in sync) or a mockgen step in
      the codegen pipeline, emitting a unifimock/unifi-mock package. Since the method list is generated, the mock is near-zero maintenance.
    - Expose a thin exported test constructor (NewClientForTesting(cfg, apiStyle) or a ClientConfig option to skip determineApiStyle) so consumers can point a real Client at httptest.NewServer without
      the controller-detection round-trip — this shares the seam proposed in TEST-09.

### TEST-11 [P2] JSON edge-case unmarshalers (numberOrString, booleanishString, emptyStringInt) and Logout/Version/Meta.error() are untested or partially covered

- **Category:** Coverage · **Effort:** M · **Risk:** Low
- **Locations:** unifi/json.go:20-40; unifi/json.go:80-91; unifi/client.go:162-174; unifi/client.go:379-389; unifi/unifi_errors.go:18-27
- **Problem:** The JSON helpers that exist precisely to handle controller edge cases are unverified: numberOrString.UnmarshalJSON (Device Channel/TxPower/StpPriority, ChannelPlan) has four branches
  all at 0%; booleanishString.UnmarshalJSON (Device LteExtAnt/LtePoe) returns a hard error for anything other than 'enabled'/'disabled' — a real failure mode no test pins; emptyStringInt is only
  81.2% (Unquote/Atoi error branches unhit). Separately, Logout (0%) has the same IsAPIKey-skip-vs-POST branch as the tested Login but is never driven; Version() (0%) caches sysInfo and swallows
  errors into ''; and Meta.error() (0%, used by CreateUser) builds a ServerError from a non-ok RC and is flagged in code comments as suspect old-style error handling that should be pinned before any
