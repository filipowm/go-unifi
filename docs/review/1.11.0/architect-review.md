# Lead Software Architect — go-unifi review (target 1.11.0)

> Generated by a two-persona multi-agent review (5 architect + 3 test-lead area reviewers, synthesized). Findings prefixed ✅ were independently re-verified against source by the lead before hand-over.

## Executive summary

**Overall health: structurally sound but carrying several latent correctness landmines.** The library has a clean, well-reasoned architecture — focused construction helpers, an interceptor
abstraction, a pluggable error handler, a single-source-of-truth codegen action catalog, and genuinely good security hardening in the download/extract pipeline (decompression-bomb caps, zip-slip
defense). The dominant systemic theme, however, is **drift between the hand-written layer and the codegen source of truth**, and it has already shipped real bugs.

**Tackle first (P0):** Three published, runtime-broken contracts. (1) `Version()` self-deadlocks any `UseLocking:true` client (re-entrant `sync.Mutex`). (2) `booleanishString` cannot decode the
`true|false` wire values its only two wired fields (`LtePoe`/`LteExtAnt`) actually carry, failing the *entire* `Device` fetch, and marshals the wrong shape. (3) Six exported `Setting*` interface
methods (`mdns`, `roaming_assistant`, `traffic_flow`) always fail with "unexpected key" because `settingFactories` drifted from the generated catalog — and the test that should have caught it compares
two hand-maintained mirrors against each other, so it passes green.

**Biggest themes:** (a) **Codegen↔hand-written drift** with no enforcing mechanism — broken settings, `SetSetting` unreachable, `DpiApp/DpiGroup` dead, ~22 pure-boilerplate wrapper files; (b) *
*Concurrency** — racy `sysInfo` cache and CSRF token, and a coarse request-serializer mutex that conflates two concerns and enables the deadlock; (c) **Error-handling robustness** — `HandleError`
discards HTTP status on non-JSON/empty bodies, `ErrNotFound` misses real 404s, and three custom JSON unmarshalers (`numberOrString`, `booleanishString`) corrupt or reject server values; (d) **Security
defaults** — TLS verification is OFF by default. Address the P0s and the drift-prevention test/registry generation first; they are small-to-medium effort with outsized payoff.

## Strengths

- Clean construction and request pipeline: newClientFromConfig delegates to focused, independently unit-tested helpers (buildHTTPClient, resolveCredentials, buildInterceptors, resolveErrorHandler);
  the ClientInterceptor abstraction cleanly models API-key vs CSRF auth and default headers as interceptors rather than hardcoded request logic.
- Strong extension points: pluggable ResponseErrorHandler, HttpRoundTripperProvider/HttpTransportCustomizer, and a swappable Logger interface (with a proper noopLogger) give advanced users escape
  hatches without forking.
- Consistent, easy-to-follow CRUD convention: private generated getX/listX methods re-exported by public GetX/ListX wrappers, with custom logic cleanly isolated in hand-written siblings (user.go,
  device.go, wlan.go).
- Well-structured error model: ServerError captures status/method/URL/code/validation details with a readable Error() string and good unit coverage; both UniFi v1 and v2 error envelope shapes are
  normalized into one taxonomy.
- Declarative config validation via go-playground tags runs before any network I/O; the Validator abstraction with NewCustomRegexValidator and lazy regex compilation is clean and reusable.
- Codegen pipeline is cleanly staged (download -> parse -> customize -> render) with a small Generatable interface; standardActions is a single source of truth shared by the generator and the
  excludeFunctions validator, preventing that particular drift.
- Genuinely good security hardening in the extract pipeline: copyWithLimit guards decompression bombs with documented per-artifact caps, and zip-slip is defended in depth via sanitizeExtractedPath
  plus an api/fields/*.json allowlist.
- Thoughtful, honest codegen comments mark known hacks (sorted-map spacer, IP-octet break semantics), and strong intent-revealing tests lock in the fragile numeric/float/IP/bool inference edge cases.
- Go 1.25 module means the pervasive range-and-take-address filter pattern is safe under per-iteration loop semantics; settingFactories correctly returns fresh non-aliased instances per call, an
  invariant explicitly guarded by tests.
- The Client is exposed as an interface with a compile-time var _ Client = &client{} assertion, aiding testability and keeping the concrete type unexported.

## Findings (detail)

### ARCH-01 [P0] Re-entrant deadlock: Version() with UseLocking=true self-deadlocks the calling goroutine ✅ *verified against source*

- **Category:** Concurrency · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/client.go:162-174; unifi/requests.go:131-135
- **Problem:** Version() acquires c.lock (client.go:166) and, while still holding it, calls GetSystemInformation() -> c.Get() -> Do() -> executeRequest(), which re-acquires the same c.lock at
  requests.go:131-132 when useLocking==true. sync.Mutex is NOT reentrant, so on any client built with UseLocking:true and an uncached sysInfo (e.g. a fresh NewBareClient), the very first Version()
  call hangs the goroutine permanently. The feature meant to make the client safe is what hard-hangs it. Verified by reading both call sites.
- **Proposals:**
    - Do NOT hold c.lock across the HTTP call in Version(): read the sysInfo cache under a lock, release it, fetch over HTTP without any lock, then re-acquire to store (double-checked).
    - Introduce a dedicated cache mutex (e.g. sysInfoMu sync.RWMutex) separate from the request-serialization lock so the two concerns never alias the same mutex — this also fixes ARCH-04 and ARCH-09
      at the root.
    - Add a regression test that constructs a UseLocking:true client against an httptest server and calls Version() to prove no deadlock.

### ARCH-02 [P0] booleanishString cannot decode the values its wired fields actually send, failing the whole Device fetch, and marshals the wrong shape ✅ *verified against source*

- **Category:** Error-handling · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/json.go:78-91; codegen/customizations.yml:476-479; codegen/v9.5.21/Device.json:166-167; unifi/device.generated.go:128-130
- **Problem:** booleanishString.UnmarshalJSON ONLY accepts "enabled"/"disabled" and returns an error for everything else (null, JSON booleans, "true"/"false"). But the only two fields wired to it (
  LtePoe, LteExtAnt in customizations.yml) have a controller field regex of "true|false" — NOT enabled/disabled. So any Device whose lte_poe/lte_ext_ant is present and non-empty fails to decode, and
  because the call is inside Device.UnmarshalJSON the ENTIRE device fetch fails. It also has no MarshalJSON, so a round-tripped value serializes as a JSON boolean the controller's own parser would
  reject. Confirmed: booleanishString is wired to exactly two fields and only matches enabled/disabled.
- **Proposals:**
    - Rewrite booleanishString.UnmarshalJSON (hand-written, in scope) to be permissive: accept null (zero value), JSON booleans true/false, and the strings "true"/"false", "enabled"/"disabled", "1"/"
      0", ""; never hard-error on an unexpected scalar.
    - Add a MarshalJSON emitting the form the controller expects (per the Device.json regex, the string "true"/"false"); alternatively rewire LtePoe/LteExtAnt to a plain *bool if the controller truly
      accepts JSON booleans on PUT.
    - Add a json_test.go round-trip table test covering null/true/false/"true"/"false"/"enabled"/"disabled"/"" to lock the regression down.
    - Trade-off: the Device.json "true|false" regex strongly suggests enabled/disabled was simply the wrong mapping; confirm the exact wire value against a live controller before finalizing the
      marshal form.

### ARCH-03 [P0] Three Setting getters/updaters are broken at runtime: settingFactories drifted from the generated setting catalog ✅ *verified against source*

- **Category:** Architecture · **Effort:** S · **Risk:** Low
- **Locations:** unifi/setting.go:20-69; unifi/setting_mdns.generated.go:19; unifi/setting_roaming_assistant.generated.go:19; unifi/setting_traffic_flow.generated.go:19; unifi/client.generated.go:
  783-787
- **Problem:** Codegen emits 43 Setting*Key constants and 43 typed GetSettingX/UpdateSettingX interface methods, but the hand-written settingFactories map registers only 40. SettingMdnsKey ("mdns"),
  SettingRoamingAssistantKey ("roaming_assistant"), and SettingTrafficFlowKey ("traffic_flow") are missing. Since every GetSettingX/UpdateSettingX routes through newFields() — which returns
  fmt.Errorf("unexpected key %q") for unregistered keys — six advertised, exported interface methods ALWAYS fail at runtime. Confirmed: the three constants exist in generated files but are absent from
  the map. Root cause is architectural: the catalog is hand-mirrored while the source of truth is codegen, with nothing tying them together.
- **Proposals:**
    - Immediately add the three missing entries to settingFactories (and the matching pins in setting_test.go expectedSettingTypes) to unbreak the published contract.
    - Then eliminate the hand-maintained map: have codegen emit the key->factory registry into a settings_registry.generated.go (it already generates the keys and typed getters, so it knows the full
      set), so it can never drift again.
    - Alternative if the registry must stay hand-written: emit a generated init() per setting file that self-registers (registerSetting(SettingMdnsKey, ...)), so adding a setting auto-registers.

### ARCH-04 [P1] Concurrency: sysInfo cache and CSRF token are racy; UseLocking conflates a cache guard with a global request serializer

- **Category:** Concurrency · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/client.go:127-132; unifi/client.go:163; unifi/client.go:172; unifi/client.go:317; unifi/interceptors.go:38-50; unifi/requests.go:71-79; unifi/requests.go:93-99;
  unifi/requests.go:131-135
- **Problem:** Merged from ARCH-A2/A3/A5: a single c.lock sync.Mutex is overloaded for two unrelated jobs and neither shared field is reliably protected. (1) c.sysInfo is read at client.go:163 with no
  lock, written at 172 under the lock, and written at 317 from NewClient with no lock — a textbook data race detectable by `go test -race`. (2) CSRFInterceptor.CSRFToken is read in InterceptRequest
  and written in InterceptResponse with no synchronization; with the default UseLocking==false the request path skips the lock entirely, so concurrent user/pass requests race on the token, risking
  stale/empty CSRF headers and 403s. (3) When UseLocking==true, c.lock is held across the entire c.http.Do round-trip (requests.go:131), fully serializing every request and killing HTTP concurrency —
  and aliasing that role onto the cache mutex is exactly what enables the ARCH-01 deadlock. The design forces an all-or-nothing choice between 'no safety' and 'no concurrency'.
- **Proposals:**
    - Protect mutable shared state with fine-grained, always-on synchronization: store sysInfo in an atomic.Pointer[SysInfo] (or behind a dedicated RWMutex, also used by the NewClient write), and
      protect CSRFToken with a sync.RWMutex/atomic.Value inside CSRFInterceptor so it is safe regardless of UseLocking.
    - Drop the coarse whole-request lock from executeRequest (net/http.Client is already goroutine-safe); if a per-controller serializer is genuinely needed, make it an explicit separately-named
      option that is NEVER held across the network round-trip in a way that can recurse.
    - Document the concurrency contract on Client/ClientConfig: which operations are goroutine-safe and what (if anything) UseLocking now buys.
    - Add -race tests: N goroutines calling Version() and N concurrent Get/Post on a user/pass client against an httptest controller.

### ARCH-05 [P1] HandleError discards HTTP status/method/URL when the error body is empty or non-JSON, and ErrNotFound silently misses real 404s

- **Category:** Error-handling · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/unifi_errors.go:11; unifi/unifi_errors.go:128-149; unifi/requests.go:101-103; unifi/firewall_group.generated.go:62-79; unifi/described_feature.go:23; unifi/setting.go:93
- **Problem:** Merged from ARCH-A10/B2/B3. (1) HandleError immediately json-decodes resp.Body and returns the raw decode error verbatim on failure. For the common cases of an empty body (401/403) or
  an HTML gateway page (502/504), the caller gets a bare `EOF` or `invalid character '<'` with the status code, method, and URL all lost — exactly when diagnostics matter most. The rich ServerError
  happy-path throws its own context away on the failure path. (2) The ErrNotFound sentinel is returned ONLY when a call SUCCEEDS (2xx) but the data array is the wrong length; a real HTTP 404 instead
  surfaces as a *ServerError{StatusCode:404}. So `errors.Is(err, ErrNotFound)` catches the empty-array case but completely misses genuine 404s, making the documented not-found signal unreliable.
- **Proposals:**
    - Read the body once into a capped []byte (io.ReadAll with a sane limit) BEFORE decoding, so the raw text is available for a fallback. On decode failure or io.EOF, still return a populated *
      ServerError{StatusCode, RequestMethod, RequestURL} with the raw body/decode error in Message instead of the bare error.
    - Map StatusCode==404 to ErrNotFound: give ServerError an Is(target) (or Unwrap) so errors.Is(err, ErrNotFound) holds for a 404 ServerError; alternatively provide an exported IsNotFound(err)
      helper that checks both ErrNotFound and StatusCode==404 and document the contract. Prefer the Is approach so existing callers work unchanged.
    - Add tests: empty body and HTML body assert a *ServerError with the correct StatusCode; assert errors.Is(err, ErrNotFound) holds for both a 404 ServerError and an empty-data 200.

### ARCH-06 [P1] TLS verification is OFF by default: VerifySSL zero-value disables certificate checks ✅ *verified against source*

- **Category:** Security · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/client.go:62-77; unifi/client.go:198
- **Problem:** VerifySSL is a plain bool whose zero value is false, and buildHTTPClient sets InsecureSkipVerify: !config.VerifySSL (client.go:198). A caller who constructs ClientConfig WITHOUT setting
  VerifySSL gets InsecureSkipVerify==true — TLS verification silently disabled and MITM-vulnerable. Secure-by-default is violated: the safe configuration requires an affirmative opt-in, and the doc
  comment buries that the default IS insecure. UniFi controllers commonly use self-signed certs, but that argues for making insecure mode explicit, not the default.
- **Proposals:**
    - Invert to a secure-by-default field, e.g. InsecureSkipVerify bool or SkipTLSVerify bool (default false => verification on); keep VerifySSL only as a deprecated alias if back-compat matters. This
      is a breaking change, so gate it on a minor/major bump and document the migration.
    - If the field name must stay for back-compat, default it to true via a constructor/normalization step requiring explicit opt-out, and emit a Warn log whenever verification is disabled.
    - Update the doc comment and README/configuration.md to state the security implication prominently.
- **Resolution (shipped):** Took proposal 1 — inverted to a secure-by-default plain bool, renamed `ClientConfig.VerifySSL bool` → `SkipVerifySSL bool` (zero value `false` ⇒ verification ON; `true` ⇒ disabled, logged at WARN). No deprecated alias kept; the rename forces a compile error at every call site so the secure-by-default flip surfaces at build time. Documented as a breaking change in [breaking_changes.md](breaking_changes.md).

### ARCH-07 [P1] numberOrString decodes JSON null to the literal string "null", corrupting values on round-trip ✅ *verified against source*

- **Category:** Error-handling · **Effort:** S · **Risk:** Low
- **Locations:** unifi/json.go:16-40
- **Problem:** numberOrString.UnmarshalJSON only special-cases the empty-string token; any other unquoted token falls through to *e = numberOrString(string(b)). A JSON null therefore becomes the
  4-character string "null", and booleans/other scalars are stored as raw text. Fields like Device.StpPriority and RadioTable.Channel/TxPower use this type; a server-sent null silently becomes a
  bogus "null" string that is then marshaled straight back to the controller on the next PUT, corrupting the value.
- **Proposals:**
    - Treat the literal null token as empty string (or leave the field zero), mirroring the existing "" handling.
    - Reject or normalize other non-numeric, non-string scalars instead of storing raw bytes.
    - Add json_test.go cases for null, integer, float, and quoted-string inputs.

### ARCH-08 [P1] Codegen<->hand-written drift is unguarded across the public surface (SYSTEMIC) ✅ *verified against source*

- **Category:** Architecture · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/setting.go:20-61; unifi/setting.go:71-106; unifi/setting_test.go:12-72; codegen/clients.go:44-93; codegen/customizations.yml:5; codegen/customizations.yml:153-165;
  unifi/client.generated.go; unifi/dpi_app.generated.go; unifi/dpi_group.generated.go
- **Problem:** Systemic theme tying together ARCH-03, plus ARCH-C-P1-1/P1-2/P2-1: the generated public Client interface is built purely from customizations.yml + the standard CRUD catalog and does NOT
  introspect the actual exported methods on *client, while several hand-maintained mirrors must be kept in lockstep with codegen by hand. Concrete failures: (a) settingFactories drifted -> 6 broken
  methods (ARCH-03); (b) SetSetting is a real exported method on *client (setting.go:71) but absent from customizations.yml AND the Client interface, so external callers cannot invoke it even though
  its read counterpart GetSetting is exposed — confirmed not present in client.generated.go; (c) DpiApp/DpiGroup are excluded from the interface but, unlike other excluded resources, get NO
  hand-written wrapper, so their full generated types and CRUD are dead, inaccessible code that still ships; (d) the setting coverage test compares two hand-maintained mirrors (settingFactories vs
  expectedSettingTypes) that are both missing the SAME 3 keys, so it can never detect codegen drift — the exact failure mode that slipped through.
- **Proposals:**
    - Add a reflection-based guard test that asserts every exported method on *client is present in the Client interface (or explicitly allow-listed as intentionally interface-private), catching
      SetSetting-style drift automatically — and decide SetSetting's intent (expose via customizations.yml to mirror GetSetting, or rename to unexported setSetting).
    - Replace the setting coverage test's hand-written expectedSettingTypes with one derived from the generated source of truth: iterate every Setting*Key constant (or every GetSettingX method) and
      require newFields() to succeed, so a new codegen setting that isn't registered fails the build immediately.
    - Generate the registry/interface from the AST of exported *client methods (or generate the key->factory registry) instead of (or in addition to) the YAML list, so neither can omit an implemented
      public method.
    - Resolve Dpi*: either drop the generated dpi_*.generated.go entirely (exclude from generation, not just the interface) if DPI is unsupported, or add hand-written wrappers + customizations.yml
      entries like FirewallZoneMatrix/DescribedFeature got. Document each excludeResources entry's intent next to it in customizations.yml.

### ARCH-09 [P2] Constructor mutates caller-owned ClientConfig (URL and UserAgent)

- **Category:** Maintainability · **Effort:** S · **Risk:** Low
- **Locations:** unifi/client.go:239-245; unifi/client.go:272
- **Problem:** newClientFromConfig writes config.URL = strings.TrimRight(...) (client.go:272) and buildInterceptors writes config.UserAgent = defaultUserAgent (client.go:242) directly onto the
  caller's *ClientConfig. Mutating a caller-supplied struct is a surprising side effect: a caller reusing one ClientConfig to build several clients, or inspecting it afterward, sees it silently
  altered. It couples the public input struct to internal normalization and makes the functions non-idempotent w.r.t. their argument.
- **Proposals:**
    - Copy the config (or extract needed values into locals / an internal normalized struct) before mutating, leaving the caller's ClientConfig untouched.
    - Compute the resolved UserAgent and trimmed URL as locals passed down rather than written back into config.
    - If the mutation is intentional, document it on ClientConfig; otherwise treat the input as read-only.

### ARCH-10 [P2] Meta.error() soft-error path (200 with rc:error) is unchecked across the entire generated CRUD surface

- **Category:** Error-handling · **Effort:** M · **Risk:** Medium
- **Locations:** unifi/unifi_errors.go:18-27; unifi/user.go:62-65; unifi/firewall_group.generated.go:62-122
- **Problem:** The UniFi v1 API can return HTTP 200 with meta.rc == "error" (a soft/application-level failure). Meta.error() exists to detect this but is called in only ONE place (user.go, itself
  marked with a TODO questioning whether it's still needed). Every generated CRUD method decodes Meta into respBody and then ignores meta.rc entirely — it only checks len(Data). So a 200 carrying
  meta.rc=error with an empty data array surfaces to the caller as the generic ErrNotFound (or nil error with empty data), masking the real server-reported message. The taxonomy is split: HandleError
  covers non-2xx, but the 2xx-rc-error case is effectively unhandled.
- **Proposals:**
    - Centralize the check: have the hand-written handleResponse in requests.go optionally decode a Meta envelope and surface a ServerError when rc==error, so every decoded {meta,data} response is
      validated; then drop the one-off user.go call and resolve its TODO. Trade-off: this couples the generic response handler to the v1 envelope shape, so guard it to only trigger when a meta block
      is actually present.
    - Alternatively, generate the Meta.error() check into the CRUD templates so all resources behave uniformly.
    - Add a test that a 200 with meta.rc=error produces a ServerError, not ErrNotFound.

### ARCH-11 [P2] ContentLength==0 short-circuit silently skips decoding bodies that lack a declared length

- **Category:** Error-handling · **Effort:** S · **Risk:** Low
- **Locations:** unifi/requests.go:104-113
- **Problem:** handleResponse returns early without decoding when resp.ContentLength == 0 (requests.go:105, confirmed). Chunked HTTP/1.1 responses report -1 (handled), but a server, proxy, or HTTP/2
  path can deliver a non-empty JSON body while reporting ContentLength==0; respBody is then silently left zero-valued and the caller sees an empty result with no error. The guard couples 'should I
  decode?' to a transport header that is not a reliable emptiness indicator — a latent correctness trap dependent on controller/proxy behavior.
- **Proposals:**
    - Decide based on the body itself, not ContentLength: attempt to decode and treat io.EOF (empty body) as 'no content' (if errors.Is(err, io.EOF) { return nil }).
    - Keep respBody==nil as the only unconditional skip; otherwise stream-decode and tolerate empty input.
    - Add a test exercising a 200 response with a JSON body but ContentLength unset/0.

### ARCH-12 [P2] ~22 resource wrapper files are 100% pure-delegation boilerplate the codegen could emit

- **Category:** DRY · **Effort:** L · **Risk:** Medium
- **Locations:** unifi/ap_group.go; unifi/broadcast_group.go; unifi/channel_plan.go; unifi/tag.go; unifi/port_forward.go; unifi/port_profile.go; unifi/user_group.go; unifi/radius_profile.go;
  unifi/dhcp_option.go; unifi/dynamic_dns.go; unifi/routing.go; unifi/firewall_group.go
- **Problem:** At least 22 hand-written files contain only mechanical 5-method blocks of the form `func (c *client) ListX(...) { return c.listX(...) }` with zero custom logic — ~110+ trivial methods
  that must be hand-maintained in lockstep with the generated private methods. Every new resource requires hand-creating yet another identical file, and any template signature change must be echoed by
  hand across all of them. The codegen already knows each resource's name and full action catalog, so it has everything needed to emit these wrappers. This is the bulkiest manifestation of the
  systemic drift theme (ARCH-08).
- **Proposals:**
    - Have codegen emit the public CRUD wrapper directly for resources flagged as having no customization (GetX/ListX/CreateX/UpdateX/DeleteX calling the private methods), falling back to a
